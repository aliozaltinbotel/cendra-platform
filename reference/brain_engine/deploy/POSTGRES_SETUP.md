# cendra-pg deployment guide

Primary PostgreSQL datastore for Brain Engine — backed by the
CloudNativePG (CNPG) operator that is already installed in the
`botelai-prod` cluster.

## What this deploys

- **`deploy/postgres-cluster.yaml`** — CNPG `Cluster` resource. Boots a
  single PostgreSQL 17 instance with `pgvector`, `pg_trgm`, and
  `uuid-ossp` extensions, a `brain_engine` database, and a
  `brain_engine` owner user. Exposes `cendra-pg-rw` (primary) service.
- **`deploy/postgres-migrations.yaml`** — ConfigMap + Job that creates
  the 7 Brain Engine tables (`decision_cases`, `pattern_rules`,
  `zfs_blocks`, `zfs_pointers`, `zfs_snapshots`, `guest_memories`,
  `interactions`) with indexes and HNSW vector indexes.

## Prerequisites (already satisfied)

- CNPG operator installed in cluster (CRDs: `postgresql.cnpg.io`).
- Default StorageClass available (AKS `managed-csi`).
- `dev` namespace exists.

## Apply sequence

### Step 1 — Create the cluster

```bash
kubectl apply -f deploy/postgres-cluster.yaml -n dev
```

Wait for the cluster to become healthy. In Lens: **Custom Resources
→ postgresql.cnpg.io → Cluster → cendra-pg** — wait for
`status.phase` to read `Cluster in healthy state` (~2-3 minutes).

From terminal:
```bash
kubectl wait --for=jsonpath='{.status.phase}'='Cluster in healthy state' \
  cluster/cendra-pg -n dev --timeout=10m
```

### Step 2 — Verify extensions

```bash
kubectl exec -it cendra-pg-1 -n dev -c postgres -- \
  psql -U brain_engine -d brain_engine \
  -c "SELECT extname, extversion FROM pg_extension ORDER BY extname;"
```

Expected output:
```
 extname  | extversion
----------+------------
 pg_trgm  | 1.6
 plpgsql  | 1.0
 uuid-ossp| 1.1
 vector   | 0.8.0
```

Image used: `ghcr.io/tensorchord/cloudnative-vectorchord:17.5-0.4.3`
(PostgreSQL 17.5 + pgvector 0.8.0 + VectorChord 0.4.3).

### Step 3 — Apply schema migrations

```bash
kubectl apply -f deploy/postgres-migrations.yaml -n dev
```

The Job creates tables and HNSW indexes. Watch progress:
```bash
kubectl logs -f job/cendra-pg-migrate-001 -n dev
```

Expected final lines:
```
             List of relations
...
 public | decision_cases | table | brain_engine
 public | pattern_rules  | table | brain_engine
 public | zfs_blocks     | table | brain_engine
 ... (7 tables total)
Migration complete.
```

### Step 4 — Wire brain-engine to the database

CNPG auto-generates the secret `cendra-pg-app` containing connection
details. Read the ready-to-use URI:

```bash
kubectl get secret cendra-pg-app -n dev \
  -o jsonpath='{.data.uri}' | base64 -d
```

The URI has this shape (password auto-generated by CNPG):
```
postgresql://brain_engine:<PASSWORD>@cendra-pg-rw.dev.svc:5432/brain_engine
```

Add it to `deploy/brain-engine-dev.yaml` under `env:`:
```yaml
env:
  ...
  DATABASE_URL: "postgresql://brain_engine:<PASSWORD>@cendra-pg-rw.dev.svc:5432/brain_engine?sslmode=require"
```

Trigger a rollout:
```bash
kubectl rollout restart deployment/brain-engine -n dev
```

Verify from inside a brain-engine pod:
```bash
kubectl exec -it deploy/brain-engine -n dev -- \
  python -c "import os; print('DATABASE_URL' in os.environ)"
```

## Scaling up (later)

When ready for HA, edit `postgres-cluster.yaml`:
```yaml
spec:
  instances: 3  # 1 primary + 2 standby
```
Apply. CNPG provisions replicas with streaming replication, no
downtime. Failover happens automatically in 10-30s on primary loss.

## Backup (deferred)

Once we start writing production data, add a `ScheduledBackup` resource
pointing to an Azure Blob Storage account. Shape:
```yaml
apiVersion: postgresql.cnpg.io/v1
kind: ScheduledBackup
metadata:
  name: cendra-pg-daily
  namespace: dev
spec:
  cluster:
    name: cendra-pg
  schedule: "0 0 2 * * *"  # 02:00 UTC daily
  backupOwnerReference: self
```
plus `.spec.backup.barmanObjectStore` on the Cluster pointing to Azure
Blob. Decide on backup strategy when we have non-trivial data volume.

## Troubleshooting

- **Pod stuck in Pending** — check PVC binding:
  `kubectl describe pvc -n dev`. If `no available persistent volumes`,
  the default StorageClass has no provisioner. Add
  `spec.storage.storageClass: managed-csi` explicitly.
- **`CREATE EXTENSION vector` fails** — the image lacks pgvector.
  Verify `imageName` in Cluster spec points to the vectorchord image.
- **Migration Job fails with `role "brain_engine" does not exist`** —
  bootstrap didn't complete; check Cluster events with
  `kubectl describe cluster cendra-pg -n dev`.
- **brain-engine still uses SQLite after restart** — `DATABASE_URL`
  not picked up. Check `env | grep DATABASE_URL` inside the pod.
